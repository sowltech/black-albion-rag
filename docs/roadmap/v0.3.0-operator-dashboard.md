# BLACK ALBION RAG v0.3.0 — Operator Dashboard

## Purpose

v0.3.0 will add a browser-accessible operator dashboard so BLACK ALBION RAG can
be used visually, not only through API endpoints. The dashboard should give an
operator a simple read-only control surface for health, retrieval inventory,
search, query testing, release status, and governance visibility.

## Current Foundation

The v0.2.0 proof establishes:

- FastAPI app boots.
- Live uvicorn smoke test passes.
- Enterprise GPT OS validation passes.
- Eval runner passes.
- Dual GitHub Actions workflows pass.
- v0.2.0 GitHub Release exists.

## Dashboard Scope

The first dashboard should include:

- System health/status panel.
- Module count and module list.
- Site count and site list.
- Claims count and claim list.
- Search box using existing search endpoint.
- Query form using existing `POST /query` endpoint.
- Links to OpenAPI docs.
- Enterprise GPT OS status summary.
- Release/version panel.
- Repo estate status link.

## Non-Goals

v0.3.0 will not yet include:

- Authentication.
- Database migration.
- Heavy frontend framework.
- Destructive admin actions.
- Paid API integration.
- Autonomous agents.
- Editing/deleting source data.

## Technical Approach

Use the least invasive implementation that fits the existing FastAPI app:

- Add simple FastAPI HTML routes/templates if the existing backend supports it.
- Or serve a lightweight static dashboard.
- Reuse existing API endpoints.
- Avoid new frontend build complexity unless necessary.

The dashboard should remain read-only for this milestone. It should display
existing data and call existing endpoints rather than introducing new mutation
paths or data models.

## Proposed Routes

- `GET /dashboard`
- `GET /dashboard/health`
- `GET /dashboard/modules`
- `GET /dashboard/sites`
- `GET /dashboard/claims`
- `GET /dashboard/search`
- `GET /dashboard/query`

## Acceptance Criteria

v0.3.0 is complete when:

- `/dashboard` loads in browser.
- Dashboard shows health status.
- Dashboard shows modules/sites/claims summary.
- Search works from browser.
- Query form works from browser.
- No existing API endpoints break.
- Smoke test updated to include `/dashboard`.
- `pytest` passes.
- Enterprise GPT OS validation passes.
- Live Uvicorn Smoke CI passes.
- CHANGELOG updated.
- Tag `v0.3.0` created after release.

## Risks

- Frontend complexity creep.
- Accidental app logic changes.
- Fragile HTML tests.
- Mixing dashboard and data mutation too early.

## Build Sequence

1. Create dashboard planning doc.
2. Inspect current FastAPI app structure.
3. Add minimal dashboard route.
4. Add dashboard template/static file.
5. Add dashboard smoke probe.
6. Add tests if appropriate.
7. Update docs.
8. Commit.
9. Push and verify CI.
10. Release/tag only after stable.

## Recommended First Implementation Step

Add a read-only `/dashboard` route that renders health, counts, and links using
existing in-memory/API data.
